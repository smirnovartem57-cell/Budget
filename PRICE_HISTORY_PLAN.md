# ТЗ / план реализации: История цен по позициям чеков («Цены»)

Статус: **проектирование** (код ещё не пишем). Отдельный экран-лист в приложении.
Дата: 2026-07-20.

## 1. Цель

Запоминать каждую позицию из чека (наименование, цена за единицу, скидка, количество,
магазин, дата), чтобы отвечать на вопросы:

- «Сколько стоило это молоко вчера / месяц / год назад?»
- «Где этот товар выгоднее — в Пятёрочке или в Ленте?»
- «Как менялась цена конкретного товара во времени?»
- «Сколько я потратил на конкретный товар/категорию за период?»

Всё это — **детерминированные расчёты на JS**. AI используется только для распознавания
текста наименований (нормализации), но **никогда не считает деньги, цены, скидки или %**.

## 2. Источник данных чека — решение: QR → ОФД, фото как запас

Российский кассовый чек содержит QR-код с фискальными реквизитами:
`t` (дата/время), `s` (сумма), `fn` (номер ФН), `i` (номер ФД), `fp` (фиск. признак),
`n` (тип). По этим реквизитам через сервис проверки чеков (ОФД / proverkacheka-подобный
API) возвращается **точный JSON со всеми позициями**: наименование, количество, цена,
сумма, скидка, магазин (ИНН/название/адрес), дата.

Почему так, а не фото+AI:
- Точные цифры прямо из ФНС — без ошибок распознавания сумм и потерянных скидок.
- Наименования приходят в том виде, как их пробил магазин (стабильный ключ для истории).

Путь:
1. **Основной:** пользователь наводит камеру на QR чека → парсим QR (клиентски) →
   Edge Function запрашивает ОФД по реквизитам → получаем позиции.
2. **Запасной (нет QR / бумага выцвела):** фото чека → AI-распознавание позиций
   (та же зависимость, что F3; помечаем такие позиции как `source: 'photo'` — менее
   надёжные, при желании пользователь правит вручную).

Зависимости: доступ к внешнему ОФД API из Supabase Edge Functions (сеть + возможный
ключ/лимиты); для запасного пути — AI-ключ (как для F3). Оба секрета живут на сервере,
не в клиенте.

## 3. Сопоставление товаров — решение: AI-нормализация имён

Чтобы «Млк Простокв 2,5%» из одного чека и «Молоко Простоквашино 2.5% 900мл» из другого
считались **одним товаром** для сравнения цен:

- Сырое наименование из чека сохраняется всегда (`raw_name`) — оно неизменно и точно.
- Отдельно AI приводит сырое имя к **нормализованному товару** (`product`): бренд, вид,
  объём/вес, единица. Похожие написания склеиваются к одной карточке товара.
- Защита от ложных склеек (AI иногда объединит разное): пользователь может вручную
  **разъединить** позицию или **объединить** две карточки. Ручное решение приоритетнее
  и запоминается (чтобы AI не переклеивал обратно).
- Нормализация выполняется на сервере (Edge Function) пачками, результат кешируется по
  `raw_name` → один и тот же текст не нормализуем повторно.

## 4. Модель данных (Supabase)

Три новые таблицы (все с `user_id` и RLS, как остальные):

**`receipts`** — шапка чека:
- `id`, `user_id`
- `purchased_at` (timestamptz — дата и время покупки из чека)
- `store_name`, `store_inn`, `store_address`
- `total` (итог чека)
- `source` — `'qr'` | `'photo'`
- фискальные: `fn`, `fd`, `fp` (для дедупликации — один чек не заводим дважды)
- `raw_json` (полный ответ ОФД, на будущее)
- `linked_transaction_id` (nullable — связь с операцией-расходом в текущей модели)
- `created_at`

**`receipt_items`** — позиции чека:
- `id`, `receipt_id`, `user_id`
- `raw_name` (как в чеке)
- `product_id` (nullable — ссылка на нормализованный товар)
- `quantity`, `unit` (шт/кг/л)
- `price` (цена за единицу), `discount` (скидка по позиции), `sum` (итог по позиции)
- `manual_product_override` (bool — пользователь зафиксировал товар вручную)
- `created_at`

**`products`** — нормализованные товары:
- `id`, `user_id`
- `canonical_name`, `brand`, `size` (объём/вес), `unit`
- `aliases` (массив сырых имён, которые к нему относятся)
- `created_at`

Дедуп чеков: уникальность по (`user_id`, `fn`, `fd`, `fp`) — повторный скан того же чека
не создаёт дубль.

## 5. Экран «Цены» (отдельный лист)

- **Поиск товара** сверху (по нормализованному имени и по сырым алиасам).
- **Карточка товара**: последняя цена, минимальная цена и **где выгоднее** (магазин),
  история «дата · магазин · цена · скидка» списком + компактный график динамики цены.
  Все значения — из `receipt_items`, чистый JS.
- **Список чеков**: дата, магазин, итог; тап → позиции чека; правка/объединение товаров.
- **Добавление чека**: кнопка «Сканировать QR» (осн.) / «Сфотографировать» (запас).
- Редактирование/детали — в bottom sheet, как во всём мобильном UI.

## 6. Связь с текущей моделью

Чек — это один расход. Варианты:
- **A (рекомендую для MVP):** чек живёт в новых таблицах отдельно; при желании линкуется к
  операции-расходу (`linked_transaction_id`) — чтобы итог чека попал в общие расходы без
  двойного счёта. Позиции — детализация «под» операцией.
- Не ломаем существующие «Операции»/«Ежемесячные»/аналитику: история цен — надстройка.

## 7. Что остаётся детерминированным (не AI)

- Любые цены, скидки, суммы, минимумы, «где выгоднее», динамика % — только JS по данным
  из ОФД/позиций.
- AI отвечает исключительно за текст: распознать позиции с фото (запасной путь) и
  нормализовать наименования к товарам.

## 8. Этапы внедрения

1. **Схема БД** (3 таблицы + RLS + дедуп-индекс).
2. **Ввод чека по QR** → Edge Function к ОФД → сохранение `receipts` + `receipt_items`
   (без нормализации — товар = raw_name).
3. **Экран «Цены»**: список чеков, карточка товара с историей и «где выгоднее»
   (на сырых именах — уже полезно).
4. **AI-нормализация** имён → таблица `products`, склейка алиасов, ручное объединение/
   разъединение.
5. **Запасной путь фото+AI** для чеков без QR.
6. Линковка чека к операции-расходу (единый учёт).
